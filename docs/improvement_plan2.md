# MultiTrust — Product Improvement Plan

**Author:** Principal Product Manager review
**Date:** 2026-04-30
**Scope:** Feature & improvement roadmap to make MultiTrust genuinely useful for engineers and individuals building (and operating) multi-agent AI systems.
**Companion to:** `simplification_plan.md` (internal code health) and `CHANGELOG.md` (shipped surface).

---

## 1. Where we are today

MultiTrust 0.1 ships a credible, well-engineered SDK:

- **Core math**: Subjective Logic (`Opinion`, fusion, discount, decay, mapping) — implemented, tested, and stable.
- **Manager surface**: `TrustManager`, `SyncTrustManager`, admin/bulk ops, snapshots, authorities, explain, timeline.
- **Storage**: in-memory, SQLite, Redis, plus an evidence ledger with the same three backends.
- **Integrations**: LangGraph + OpenAI Agents (Tier 1), CrewAI/Google ADK/Anthropic (experimental), MCP (protocol).
- **Observability**: EventBus, Prometheus metrics, structlog, OpenTelemetry, structured `explain_trust()`.
- **Evaluation harness**: declarative scenario corpus, deterministic runner, JSON/markdown reporters, diffs.

This is a strong technical foundation. The product gap is **distribution and adoption surface**: the math is correct, but only Python developers willing to embed the library can taste it. There is one example, no service mode, no dashboard, no CLI, and no published docs site. Subjective Logic's competitive advantage (modeling *uncertainty* as a first-class quantity) is currently hidden behind a scalar `trustworthiness` projection.

This plan is a **product roadmap** — not a refactoring plan — organized by who benefits and when.

## 2. Target personas

We segment the market into four personas. Every feature below is tagged with the personas it primarily serves.

| Tag | Persona                              | What they need                                                                 |
|-----|--------------------------------------|--------------------------------------------------------------------------------|
| **DEV**  | Application developer (solo / small team) building a multi-agent app | 5-minute install-to-value; copy-paste examples; framework adapters; zero ops   |
| **PLAT** | Platform / infra engineer at a mid-large company running agents in prod | HA storage, RBAC, multi-tenancy, deploy artifacts, SLO-grade observability    |
| **SAFE** | AI safety / trust & safety / red-team engineer | Drift detection, anomaly alerts, eval harness extensions, adversarial robustness |
| **OPS**  | Operator / on-call running a deployed system   | Dashboard, CLI, runbooks, audit trail, snapshot/restore, what-if simulation    |

## 3. Themes & roadmap structure

Six product themes, each independently fundable:

1. **Adoption & Onboarding** — make the first hour delightful.
2. **Service & Deployment** — let teams run MultiTrust without embedding Python.
3. **Framework Reach** — meet developers where their agents already live.
4. **Trust Intelligence** — make the math do more for safety and analysis.
5. **Operator Experience** — turn audit trails into actionable workflows.
6. **Ecosystem & Governance** — sustain the project beyond the initial author.

Each feature is rated:

- **Effort**: S (≤1 wk), M (1–4 wk), L (1–3 mo), XL (>3 mo)
- **Impact**: ★ low, ★★ medium, ★★★ high
- **Wave**: NOW (next release, weeks 0–8), NEXT (months 2–6), LATER (months 6–12+)

## 4. Theme 1 — Adoption & Onboarding

**Why it matters:** A developer who can't get a working example running in 10 minutes never becomes a user. Today there is exactly one example (`hallucination_firewall.py`) and no published documentation site. PyPI says *0.1.0 alpha* but the unreleased CHANGELOG already has substantial new surface area — that gap erodes trust before the user has tried anything.

### 4.1 Documentation site (DEV, OPS) — **NOW, M, ★★★**
- Stand up `mkdocs-material` (or Docusaurus) at `multitrust.dev` (or GH Pages).
- Sections: Quickstart, Concepts (with the Subjective Logic visual primer), API reference (autogenerated from docstrings), Cookbook, Integrations, Operator Guide.
- Auto-publish on tag from CI.
- **Acceptance:** A developer can find "how do I gate a LangGraph node on trust?" in under 30 seconds via search.

### 4.2 Examples gallery (DEV) — **NOW, M, ★★★**
Expand from 1 to ~8 worked examples, each ≤200 LOC and runnable with `uv run python`:
- Hallucination firewall *(exists)*.
- Tool-use abuse gating (agent over-calls expensive tools → trust drops).
- Multi-source consensus voting with discounting.
- Routing: pick the most-trusted agent for a task (`rank_agents`).
- Authority federation: two services share a Redis-backed trust store.
- Decay-driven re-onboarding: agents must "earn back" trust after staleness.
- A/B comparison of two `MultiTrustConfig`s on the same workload.
- MCP server example: expose trust to an external agent via stdio.

### 4.3 Interactive Colab / Jupyter notebooks (DEV, SAFE) — **NOW, S, ★★**
Two notebooks: "Subjective Logic in 15 minutes" and "Build a trust-gated pipeline." `mybinder.org`–compatible. Linked from the README.

### 4.4 5-minute video walkthrough (DEV) — **NEXT, S, ★★**
Loom or YouTube; embed in README and docs landing.

### 4.5 Concept primer: "What's wrong with reputation scores?" (DEV, SAFE) — **NOW, S, ★★★**
A standalone Markdown page in `/doc` (and on the docs site) that explains why uncertainty matters with one concrete worked example. This is the primary *differentiation* asset and currently does not exist.

### 4.6 Versioning hygiene (DEV) — **NOW, S, ★★**
- Cut a real `0.2.0` that includes everything in `[Unreleased]`.
- Move from `Development Status :: 3 - Alpha` → `4 - Beta` once 0.2.0 ships.
- Add a "Stability" badge per public symbol in API docs.

### 4.7 Comparison matrix (DEV, PLAT) — **NEXT, S, ★★**
A docs page comparing MultiTrust to NeMo Guardrails, Guardrails AI, LangChain's trust callbacks, and "roll your own scoring" — honest tradeoffs, not marketing. Earns trust with skeptical readers.

## 5. Theme 2 — Service & Deployment

**Why it matters:** Today, using MultiTrust requires Python in your application process. That excludes Node/TypeScript agent stacks, polyglot platforms, and any team whose deployment story can't accept embedding a stateful library. MCP partially helps but is agent-tool-shaped, not service-shaped.

### 5.1 HTTP service (`multitrust serve`) (PLAT, DEV) — **NOW, M, ★★★**
- FastAPI server exposing the full `TrustManager` surface as REST endpoints.
- OpenAPI schema published; generates clients in any language.
- Built-in healthcheck, `/metrics` (Prometheus), graceful shutdown.
- Backed by any configured store (default in-memory; recommend SQLite or Redis for prod).
- **Acceptance:** A Node app can call `POST /agents/{id}/evidence` and `GET /agents/{id}/trust` without touching Python.

### 5.2 gRPC service (PLAT) — **NEXT, M, ★★**
For high-throughput / streaming evidence use cases. Same surface as REST; bidi-stream variant of `submit_evidence`.

### 5.3 Official Docker image + Helm chart (PLAT, OPS) — **NOW, S, ★★★**
- Multi-arch image on ghcr.io: `ghcr.io/nobelk/multitrust:<version>`.
- Helm chart with values for storage backend, replicas, ingress, service-monitor.
- Terraform module for AWS/GCP one-click deploy. Optional, NEXT.

### 5.4 PostgreSQL store (PLAT) — **NOW, M, ★★★**
SQLite is fine for single-process / dev; Redis is great for cache-shaped workloads; but most prod teams want **Postgres** for durability + transactions + familiar ops. Implement `PostgresTrustStore` and `PostgresEvidenceLedger` using `asyncpg`. Expose the same `TrustStore` Protocol.

### 5.5 Cloud KV stores (PLAT) — **NEXT, M, ★★**
`DynamoDBTrustStore`, `FirestoreTrustStore`, optional Cassandra. Each as an extra; all opt-in.

### 5.6 Streaming evidence ingestion (PLAT, SAFE) — **NEXT, M, ★★**
- Kafka and NATS consumers that read evidence off topics and call `submit_batch`.
- Idempotency via evidence-id dedup.
- Backpressure-aware; batches with a configurable flush window.
- **Why:** Evidence often comes from log pipelines, not RPC. This unlocks "tail your logs into MultiTrust."

### 5.7 Background scheduler (OPS, PLAT) — **NOW, S, ★★★**
Today, `apply_decay()` only runs when called. Add an optional scheduler thread/task that calls `apply_decay()` on an interval (configurable). Same hook can later run snapshot exports, ledger compaction, and stale-agent eviction. Off by default; one config flag turns it on.

### 5.8 Multi-tenancy / namespacing (PLAT) — **NEXT, L, ★★**
First-class tenant scoping (`tenant_id` everywhere there is currently `agent_id`). Storage backends gain a per-tenant index. Snapshots become tenant-scoped. RBAC (5.9) keys off tenant.

### 5.9 Authentication & RBAC (PLAT, OPS) — **NEXT, L, ★★★**
- API tokens for the HTTP service with scopes: `evidence:write`, `trust:read`, `admin:*`.
- Authority identity is signed (HMAC or Ed25519) so `submit_evidence` from a forged authority is rejected.
- Admin API requires elevated scope; `actor_id` is derived from the token, not free-form.
- **Why:** Today the SDK trusts callers implicitly. In a hosted/multi-team setup this is a hard blocker.

## 6. Theme 3 — Framework Reach

**Why it matters:** Developers adopt trust frameworks **inside** the agent framework they already use. The current Tier 1 list (LangGraph + OpenAI Agents) is a credible start, but real adoption requires the long tail.

### 6.1 Promote Anthropic to Tier 1 (DEV) — **NOW, S, ★★★**
Add contract tests, document the supported `anthropic` SDK version range. Anthropic is Claude's home; this should not be experimental.

### 6.2 Promote CrewAI to Tier 1 (DEV) — **NEXT, S, ★★**
Same playbook: contract tests + docs + a worked example.

### 6.3 New integration: AutoGen / Microsoft AG2 (DEV) — **NEXT, M, ★★**
AutoGen is one of the highest-traffic multi-agent frameworks. Worth a dedicated adapter (`GroupChatManager` hook + per-agent gating).

### 6.4 New integration: Pydantic AI (DEV) — **NEXT, S, ★★**
Small, growing fast, type-first — natural fit for MultiTrust's typed core.

### 6.5 New integration: LlamaIndex AgentWorkflow (DEV) — **NEXT, M, ★★**
Same gating + evidence-collection pattern as LangGraph nodes.

### 6.6 New integration: Plain LangChain (non-graph) (DEV) — **NEXT, S, ★★**
Many users are still on classic LangChain. A `TrustCallbackHandler` lets them adopt without migrating to LangGraph.

### 6.7 JS/TS client SDK (DEV) — **LATER, L, ★★**
Once the HTTP service (5.1) exists, ship `@multitrust/client` on npm. Read-mostly surface first (`getTrust`, `submitEvidence`); admin later.

### 6.8 Browser collector (SAFE, DEV) — **LATER, M, ★**
Tiny ESM module that batches browser-side observations (e.g., user-correction signals from a chat UI) and posts them as evidence. Useful for human-in-the-loop trust signals.

## 7. Theme 4 — Trust Intelligence

**Why it matters:** This is where MultiTrust beats "score in a dict." The math is here; the *features built on the math* are not. These features turn Subjective Logic from an internal implementation detail into observable user value.

### 7.1 Multi-dimensional trust (SAFE, DEV) — **NEXT, L, ★★★**
Today, an agent has one opinion. In practice, an agent might be highly trustworthy on **factuality** but unreliable on **latency** or **cost**. Add named trust *dimensions* per agent:
```python
await manager.submit_evidence(Evidence(..., dimension="factuality"))
trust = await manager.get_trust("agent-1", dimension="factuality")
```
- Backwards-compatible: default dimension when none specified.
- Decision policies can require thresholds across multiple dimensions.
- Storage: extend `TrustRecord` with a `dimensions: dict[str, Opinion]` field.

### 7.2 Contextual / per-task-type trust (SAFE) — **NEXT, M, ★★**
Like 7.1 but indexed by task type (`"summarize"`, `"code-gen"`) rather than dimension. Same storage shape; orthogonal index.

### 7.3 Drift & anomaly detection (SAFE, OPS) — **NEXT, M, ★★★**
- A built-in `DriftDetector` that emits an event when an agent's trust changes faster than a baseline (e.g., 3σ drop in 1h).
- A `UncertaintySpike` detector for when uncertainty rises sharply (model is confused, not yet wrong).
- Both publish to `EventBus` so existing alerting pipelines pick them up.
- **Why:** Subjective Logic's `uncertainty` term is unique signal — most reputation systems can't surface "I'm losing confidence in this agent" before performance degrades.

### 7.4 Counterfactual / what-if analysis (OPS, SAFE) — **NEXT, M, ★★**
- `manager.simulate(threshold=0.7, horizon_hours=24)` returns: which decisions would have flipped, which agents would have been gated.
- Pairs with the evaluation harness — operators tune thresholds against historical data, not prod traffic.

### 7.5 Sybil / collusion resistance (SAFE) — **LATER, L, ★★**
Detect when many "new" authorities suddenly vouch for one agent. Implement a `discount_by_authority_age()` policy and an alert when authority diversity for an agent drops.

### 7.6 Cost-weighted trust (SAFE, OPS) — **NEXT, S, ★★**
Add an optional `cost` field to `Evidence`. The decision policy can require higher trust for higher-cost actions ("only highly trusted agents can call paid tools"). Cheap, high-leverage feature.

### 7.7 Built-in safety rules (SAFE, DEV) — **NEXT, M, ★★★**
Today's `evidence/builtin/` covers task completion, response quality, latency, consensus. Add:
- **Prompt-injection detector rule** — flags evidence on suspicious agent inputs/outputs.
- **PII / secret leak rule** — negative evidence when output matches secret regexes.
- **Tool-use abuse rule** — over-frequency or out-of-scope tool calls.
- **Schema-conformance rule** — JSON outputs failing schema → negative evidence.
Each rule is opt-in; each ships with tests and a worked example.

### 7.8 Cryptographic provenance for evidence (SAFE, PLAT) — **LATER, M, ★★**
Sign each `Evidence` with the authority's key; ledger verifies on append. Lets a post-incident auditor prove evidence wasn't tampered with. Pairs with 5.9 (RBAC).

### 7.9 Evaluation harness extensions (SAFE) — **NEXT, M, ★★**
- **Hyperparameter sweep**: grid/random search over `trust_threshold`, `decay_half_life_seconds`, `default_prior_weight` against a corpus.
- **Config A/B**: run two `MultiTrustConfig`s against the same corpus, diff the decisions.
- **Trace ingestion**: import LangSmith / OpenAI eval traces as `EvidenceStep` sequences.
- **Promptfoo / Inspect adapters**: emit MultiTrust expectations from existing eval suites.

## 8. Theme 5 — Operator Experience

**Why it matters:** Once a team deploys MultiTrust, they need to *operate* it. Today the only operator-facing surface is the admin Python API and an in-process evidence ledger query. That's enough for one engineer running it locally; it is not enough for an on-call rotation.

### 8.1 Web dashboard (OPS) — **NEXT, L, ★★★**
- Read-only first: agent list with current trust/uncertainty, timeline charts, recent evidence, decision log.
- Bundled with the HTTP service (5.1) under `/ui`.
- Stack: React + Recharts (or similar). Ship as a single static bundle.
- Stretch: live updates via SSE off the EventBus.

### 8.2 CLI (`multitrust` command) (OPS, DEV) — **NOW, M, ★★★**
A first-class CLI installable via the package:
```
multitrust agents list
multitrust agents show <id>
multitrust agents trust <id> [--threshold 0.7]
multitrust evidence submit <id> --positive 3 --negative 1 --authority orchestrator
multitrust admin reset <id> --reason "post-incident"
multitrust snapshot export ./snap.json
multitrust snapshot import ./snap.json --mode merge
multitrust eval run path/to/corpus.yaml
multitrust serve  # starts the HTTP service
```
- Uses `click` or `typer` (no new runtime dep if optional).
- Connects either in-process (default), to a service URL, or to a configured store directly.
- **Why:** The number-one quality-of-life improvement for any persona that isn't writing Python every day.

### 8.3 Grafana dashboards (OPS) — **NEXT, S, ★★**
Ship JSON dashboards in `dashboards/` for the existing Prometheus metrics. Trust over time, evidence rate, decision rate, threshold-crossing rate, p99 explain latency.

### 8.4 Alert rule templates (OPS) — **NEXT, S, ★★**
Prometheus alert rules in `alerts/`: "agent trust dropped >X in Y minutes", "uncertainty rising for >N agents", "evidence ingest stalled". Pair with 7.3.

### 8.5 Notification sinks (OPS) — **NEXT, M, ★★**
EventBus subscribers that post to Slack, PagerDuty, generic webhook, email when threshold-crossed or drift events fire. Each is opt-in; each ≤100 LOC.

### 8.6 Snapshot dry-run + diff (OPS) — **NEXT, S, ★★**
`import_snapshot(snap, dry_run=True)` returns a diff (added / changed / removed records) without mutating state. Critical for prod-ready import flows.

### 8.7 Bulk historical import (OPS) — **NEXT, S, ★★**
CLI subcommand to import historical evidence from CSV/JSON/Parquet into a fresh store, so teams can bootstrap MultiTrust against existing eval logs without rewriting their pipeline.

### 8.8 Runbooks (OPS) — **NOW, S, ★★**
Doc-site section: "agent trust is unexpectedly low — what to check", "uncertainty climbing without negative evidence", "decay not running", "ledger growing unbounded — how to compact". Most are one page each.

### 8.9 Ledger compaction & retention (PLAT, OPS) — **NEXT, M, ★★**
Today the ledger is append-only with no retention policy beyond `InMemoryEvidenceLedger.max_size`. Add a `compact(older_than=...)` method and a scheduled task that aggregates old per-evidence entries into rolled-up daily summaries while preserving auditability.

## 11. What we are explicitly *not* doing (and why)

- **Replacing the dataclass + Protocol architecture with Pydantic / attrs.** Adds a runtime dep, breaks `mypy --strict` ergonomics, and the simplification plan already addresses the only legitimate complaint (some `__post_init__` boilerplate).
- **Splitting `trust_manager.py` into multiple files.** Discussed and rejected in the simplification plan; the lock + event-bus invariants live there cohesively.
- **Building a hosted SaaS in this repo.** Out of scope; the HTTP service (5.1) + Docker image (5.3) make a hosted offering trivial to layer on top later, in a separate repo.
- **Pivoting to a different trust formalism.** Subjective Logic is the differentiation; abandoning it would also abandon the moat.
- **Bundling an LLM judge.** Built-in safety rules (7.7) are deterministic. LLM-judge rules can ship as an *optional* extra later if there's demand, but should never be the default.

## 12. Risks and dependencies

| Risk | Mitigation |
|---|---|
| **Service mode (5.1) becomes the de-facto deployment but core SDK stagnates.** | Treat the HTTP service as a *thin* layer over the same `TrustManager`. Forbid service-only features. |
| **Multi-dimensional trust (7.1) breaks the single-opinion mental model.** | Default dimension preserves all existing call signatures. Document the upgrade path. Reserve a release for the migration. |
| **Auth & RBAC (5.9) creates two security models (embedded SDK vs. service).** | Keep the SDK trust-the-caller. Auth lives in the service layer only. Document the boundary clearly. |
| **Tier 1 promotion (6.1, 6.2) creates a maintenance treadmill if framework versions churn.** | Pin minimum versions; run weekly upstream-version smoke tests in CI; downgrade to experimental if the upstream is abandoned. |
| **Dashboard (8.1) becomes a long-running JS project that nobody owns.** | Ship the smallest viable read-only dashboard first; keep it as a static bundle; resist scope creep into editing. Defer write-actions to the CLI. |
| **Multi-tenancy (5.8) leaks across tenant boundaries.** | Storage Protocol gains a tenant-aware variant; add cross-tenant fuzz tests in CI. |

## 14. Open questions for product review

1. **Hosted service business model**: do we want to keep MultiTrust pure-OSS, or reserve the door for a managed offering? Affects how we license the dashboard and service-mode code.
2. **LLM-judge built-in rules**: yes/no/later? Strong opinions either way are fine; we just need to commit so the safety-rules roadmap (7.7) doesn't drift.
3. **Telemetry**: should the HTTP service phone home (opt-in, anonymous) so we can size the user base? Strong privacy implications either way.
4. **Trademark / project name**: is "MultiTrust" worth registering? Cheap insurance if the project gains traction.
5. **Single-author bus factor**: which 2–3 contributors do we want to actively recruit toward maintainer status, and what's the offer?

---

*End of plan. The themes above are independently fundable; the waves above are independently shippable. Nothing here requires Wave 1 to be complete before Wave 2 begins on the highest-impact items (notably 7.1, 7.3, 8.1), but they will land more cleanly if 5.1 and 8.2 ship first.*
